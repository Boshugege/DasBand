# DASBand 多目标步态追踪系统 (MOT) 设计与架构文档

## 1. 核心困境及解答：这部分工作该交由模型还是后处理？

在从“单目标（SOT）”跨越到“多目标（MOT）”时，最核心的哲学分歧是：**追踪的上下文（Track ID, 生命周期）应该在神经网络内部推断，还是在网络外部的后处理中进行？**

**结论与分工：【模型负责“高清目标检测”(Detection)，后处理负责“四元状态追踪”(Association)】**

*   **模型(U-Net)的职责 —— 兼容多态但不越权：**
    *   模型**不负责**分配 ID（因为在 DAS 数据中，纯声学条纹交叠时，仅凭局部像素根本无法判别这是“张三”还是“李四”）。
    *   模型**必须能够**同时输出同帧内的多个相互独立的高概率能量带（Multi-modal peaks）。
    *   **为了兼容 MOT，模型要做出的改变**：伪标签生成时（`pseudo_label.py`），绝不能将多人合并成单物质心；训练时要能够输出极为锐利的边缘（Sharp Boundaries），防止两个靠得很近的目标发生“黏连（Blob Merging）”。

*   **后处理的职责 —— “数据关联（Data Association）”与状态机：**
    *   在获得了模型的逐帧独立概率图（Mask）后，后处理接管一切时间维度上的“逻辑连接”。
    *   基于卡尔曼滤波（运动学短时预测） + 匈牙利算法（目标匹配） + 你的“四元状态机”（生命周期管理）构建一个 **DAS-SORT** (Simple Online and Realtime Tracking for DAS)。

## 2. 核心架构设计：四元状态机 (4-State Track Lifecycle)

为每一条潜在的“脚步轨迹（Tracklet）”维护如下四个状态：

1.  **`INIT` (出现/未知状态 - Tentative)**
    *   **触发条件**：在某一帧探测到了一个游离的概率波峰（Peak），且无法与任何现有的主流轨迹匹配上。
    *   **行为**：分配一个临时 ID，建立卡尔曼滤波器，但不立刻画在图中（防止这只是个一次性的环境突发噪音）。
    *   **转移**：连续 $N$ 帧（例如连续 3 帧）都成功在这个位置附近检测到波峰 $\rightarrow$ 转移为 `CONFIRMED`；如果在前两帧又没探测到 $\rightarrow$ 直接丢弃。

2.  **`CONFIRMED` (确认/追踪中 - Confirmed)**
    *   **触发条件**：从 `INIT` 转正，代表这是一个货真价实的物理活动目标。
    *   **行为**：卡尔曼滤波器在每次匹配到新 Peak 时执行受迫更新。图上画出实线。
    *   **转移**：如果在某一帧检测不到目标（没有波峰落入其预测邻域） $\rightarrow$ 转移为 `LOST`。

3.  **`LOST` (暂离/丢失 - Coasting / Temporary Lost)**
    *   **触发条件**：原本 `CONFIRMED` 的目标突然失去探测（可能是由于人停下来了、脚步极轻或被巨大噪音掩盖）。
    *   **行为**：**绝对不在此刻的图上画瞎编的观测线**。卡尔曼滤波器进入“纯物理预测”阶段，速度附带阻尼（Damping），且协方差矩阵（容错半径）不断膨胀。
    *   **转移**：
        *   **重连 (Reconnection)**：如果在容忍时间段内（例如 1-2 秒），在它膨胀的搜索半径里重新发现了波峰，则 $\rightarrow$ 切回 `CONFIRMED`。**并且利用类似于 RTS 평滑 的机制，将 `LOST` 期间丢失的路径通过两端端点进行平滑插值补齐（如果业务需要）。**
        *   **超时**：如果持续处于 `LOST` 状态超过阈值 $M$ 帧（例如 3 秒） $\rightarrow$ 转移为 `DEAD`。

4.  **`DEAD` (消失/销毁 - Deleted)**
    *   **触发条件**：长时间未见踪影的目标。目标彻底走出了监控区域，或在此长久停滞。
    *   **行为**：清理内存，释放该 Track ID。

## 3. 实现的数据流管道 (Pipeline Flow)

```text
[ 1. 神经网络推理 ]
输入 DAS Frame 
-> DASBandUNet 
-> 概率矩阵 Mask [T, C]  (模型兼容多目标的输出，不被强行Argmax摧毁)

        | (逐帧 Frame by Frame 下滑)
        v

[ 2. 独立波峰提取 (Peak Extraction) ]
使用 scipy.signal.find_peaks() 在当前帧找到所有局部概率极大值点 : P_1, P_2 ... P_K

        | 
        v

[ 3. 欧氏距离与马氏距离计算 (Cost Matrix) ]
计算已有的 Active Tracks (卡尔曼预测的下一时刻位置)与新测到的 Peak 之间的距离代价矩阵。

        |
        v

[ 4. 匈牙利匹配 (Hungarian Algorithm / Linear Sum Assignment) ]
将 Track 与 Peak 进行 1对1 最优匹配。
  - 匹配成功的 Track -> 更新卡尔曼状态 -> 保持/进入 CONFIRMED。
  - 没被分配到 Peak 的 Track -> 变为 LOST 或 DEAD。
  - 没被分配给旧 Track 的 新 Peak -> 成立新的 INIT Track。

        |
        v

[ 5. 平滑与出图 (Visualization) ]
只将曾经达到过 CONFIRMED 状态的轨迹线导出并绘制，多条轨迹用不同颜色区分。
```

## 4. 模型的配合与调整需求

尽管我们将连线的工作交给了代码（后处理），但这并不代表模型可以坐享其成。目前原有的代码结构在以下两点存在缺陷，需要做适配性改造以支持上述架构：

1.  **标签制作的纯粹性 (`pseudo_label.py`)**
    *   原先可能只注重将信号引向一条主要通道。在真实的多人数据中，如果是两人相向而行，原始候选点如果提取不当，会把 `centerline` 拉成了交叉的混沌线。
    *   **模型需求**：模型需要训练成一个“见坑填土”的机器。只要物理上有相干波能（对应某个人的步伐），即使信号比较弱，它也应该如实输出一个局部小峰。

2.  **后处理模块的重构 (`decoder.py` & `pipeline.py`)**
    *   彻底废弃基于“唯一输出数组 (1D Array)”的 `weighted_centroid` 和 `kalman_smooth_track`。
    *   拥抱面向对象编程（OOP）。建立一个 `class Track:`，内部自带该个体的 `id`、`state`、`kalman_filter` 以及 `history` 数组。
    *   输出不再是 `df["kalman_path_channel"]` 这单一列，而是转变为“帧号(Frame)、轨迹ID(TrackID)、通道位置(Channel)”的长表结构，兼容并发轨迹。
